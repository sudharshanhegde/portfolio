Dbms normalization 1NF,2NF and 3NF

March 26, 2025

day 3 dbms

dbmsnormalization1nf2nf3nf

Database are the backbone of the modern applications it stores everything from user profiles to transaction histories. poor database design can have duplicated data,inconsistent updates and inefficient storage.To avoid that we will use normalization.

Normalization fixes:- 1.data redundancy:- data repetition will waste the space and causes inconsistencies. 2.update anomalies:- editing data in multiple places can lead to errors. 3.Insertion/deletion issues:-adding or removing data might erase other information.

First Normal form (1NF):-

Each column must contain atomic values which means they should be indivisible. Each row must be having a primary key which is used for unique identification. There should be no repeating groups of columns.

Lets say there is an order table where order id customer and phones are there if the phone numbers are written by using commas then we wont be able to understand the difference and it would be difficult for us querying individual phone numbers.. SO what we will do is we will split phones into its atomic values such that we will store different instance of the phone number in such a way that there is not any ambiguity.

Lets say a person has 2 phone numbers and you have saved both phone numbers in a single contact then what will happen is we wont be able to know which contact we have contacted to avoid these we will save the numbers with the same name in the seperate rows so that its easier to identify.

Second normal form(2NF):-

It should be in first normal form. There should be no partial dependencies means all non key columns should depend entirely on the primary key.

lets say there is a composite primary key named orderid and productId for the table containing orderId productID productName and quantity..So the non key value which is product Id will only be dependent on the product ID which will cause the partial dependency.

The solution to these is we will split into two tables in such a way that one table just contains orderId productId and quantity while the second table contains productId and productName these will remove the partial dependency out of the question.

Third Normal form:-

Table should be in 2NF and another thing is that there should be no transitive dependencies which means that non key columns should not depend on other non key columns..

Lets consider a customer table in which lets say we have customerID city state Name of the customer and zipcode of the city what happens is customerID is not related to city and state.

solution to these will be we will lets say we will create two table in one we will store customer id name and pincode and in another table i will store pincode city and state these will help me make all the non key attributes depend completely on primary key.

These is it about 1NF,2NF and 3NF.

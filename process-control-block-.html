Understanding Process Control Block, Context Switching, and Kernel/User Modes in Operating Systems

In this blog, we’ll go over the Process Control Block (PCB), how context switching works, and the difference between kernel mode and user mode. These are all core concepts in operating systems and crucial for understanding how multitasking and process management work behind the scenes.

Process Control Block (PCB)

A Process Control Block is a data structure maintained by the operating system to keep track of information about a process.

Components of a PCB:

  1. Process State Shows whether the process is in ready, running, waiting, or terminated state.

  2. Process Identification A unique ID is assigned to each process to distinguish it from others.

  3. Program Counter Stores the address of the next instruction to execute. This ensures the process resumes correctly after being paused or interrupted.

  4. CPU Registers Includes accumulators, index registers, stack pointers, general-purpose registers, and condition codes. These are saved during interrupts to allow proper continuation.

  5. Memory Management Information Contains details like base and limit registers or page tables to ensure the process accesses only its allocated memory.

  6. Accounting and Scheduling Information Holds CPU usage, priority levels, and how long the process has been waiting. This helps the OS schedule tasks efficiently.

  7. I/O Status Information Tracks input/output operations, open files, and the I/O devices being used by the process.

Why PCB is Important:

  • Enables context switching
  • Aids in scheduling decisions
  • Supports error handling
  • Manages resource allocation

Context Switching in Operating Systems

Context refers to all the information the CPU needs to execute a process. Context switching is the process of storing the state of a currently running process and loading the state of another process.

This allows multiple processes to run seemingly at the same time—essential for multitasking and efficient CPU use.

What Happens During Context Switching:

  1. Interrupt or System Call A process is interrupted (like I/O completion or a timer interrupt), and the OS takes control.

  2. Saving Current Process State The OS saves all relevant data (like CPU registers and program counter) into the process’s PCB.

  3. Loading the Next Process State The OS loads the context of the next process from its PCB into the CPU.

  4. Resuming Execution The CPU starts or resumes executing the new process.

Key Components Involved:

  • Process Control Block
  • CPU Registers (saved/restored)
  • Interrupts
  • Scheduler (decides the next process)

Advantages of Context Switching:

  • Improves CPU utilization
  • Enables multitasking
  • Ensures fairness among processes
  • Provides flexibility

Disadvantages:

  • Introduces overhead
  • Can reduce performance if switches are too frequent
  • Adds complexity in OS design and management

Kernel Mode vs User Mode

Operating systems have dual mode operations—user mode and kernel mode—to protect the system and manage resources effectively.

User Mode:

  • The CPU runs applications initiated by the user.
  • Has limited access to hardware and system resources.
  • Applications are isolated; errors in one don’t crash the whole system.
  • Relies on system calls to interact with the OS.

Kernel Mode:

  • Also known as system mode, privileged mode, or supervisor mode.
  • OS core runs here with full access to all resources.
  • Handles tasks like process scheduling, memory management, and hardware communication.
  • Errors here can crash the entire system because of the level of access.

Mode Switching: How and Why It Happens

Events that Trigger a Mode Change:

  1. System Calls
  2. Interrupts
  3. Faults

What Happens During Mode Change:

  • The CPU switches mode (controlled by a mode bit in the CPU).
  • In kernel mode, the OS executes critical tasks.
  • After completion, the CPU switches back to user mode and resumes the user process.

Dual Mode Operations

  • Mode Bit: Indicates whether the CPU is in kernel mode (0) or user mode (1).
  • On system boot, the hardware starts in kernel mode and loads the OS.
  • When a user application runs, it operates in user mode.
  • If a trap or interrupt occurs, the CPU switches to kernel mode to handle it.
  • After processing, it switches back to user mode to resume the application.
